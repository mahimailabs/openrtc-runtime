# Per-tenant isolation is here, and the agency rung is open

> Draft announcement for OpenRTC v0.5 (v0.16.0). Copy for the OpenRTC Cloud post.

If you run voice agents for more than one client, you have felt the tension: one
shared worker is cheap but risky (one client's bad call can hurt the others), and
one worker per client is safe but expensive. OpenRTC v0.5 removes the tradeoff.

**Run every client in one pool, isolated.** A tenant is just the `tenant` key in
your dispatch metadata. From there:

- **Per-tenant provider keys.** Each client runs on its own STT/LLM/TTS keys and
  models, so you can attribute (and bill) API cost per client. Keys are never
  shared across tenants.
- **Per-tenant budgets.** Give each client a session cap. One client at capacity
  is rejected; the others keep accepting.
- **Blast-radius isolation.** If a client's calls start failing, its circuit
  breaker opens automatically, confines the damage for a cooldown, and recovers on
  its own. The healthy clients never notice.
- **Per-tenant introspection.** `openrtc top --tenant acme`, tenant-scoped logs,
  and per-tenant counts in `runtime_snapshot()`. Cost and quality attribution flow
  to voicegateway with no extra config, keyed off the tenant you already emit.

It is a few lines to adopt:

```python
pool = AgentPool(
    agent=SupportAgent,
    tenant_config={"acme": {"llm": openai.LLM(api_key="acme-key")}, ...},
    max_sessions_per_tenant={"acme": 50},
    enable_tenant_circuit_breaker=True,
)
```

Single-tenant deployments are unchanged: no `tenant` means the `"default"` tenant,
and nothing about your setup changes.

One honest note: coroutine mode shares a process, so this is trusted-multi-client
isolation, not an OS sandbox. For a hard compliance wall, run a worker per tenant.
The [multi-tenancy guide](/concepts/multi-tenancy) draws the line precisely.

The agency rung is open. Read the [guide](/concepts/multi-tenancy), then
[onboard your first tenant](/runbooks/onboarding-a-tenant).
